# 1. Crear la carpeta ciclo/

# 2. Crear archivo main.tf

```tf
resource "local_file" "file" {
  filename        = var.filename
  file_permission = "0700"
  content         = "Hello World"
}
```

# 3. Crear variables.tf

```tf
variable "filename" {
  type    = string
  default = "hello.txt"
}
```

# 4. Inicializar y Aplicar

```bash
terraform init
terraform apply
```

Salida esperada del plan:
```vim
Plan: 1 to add, 0 to change, 0 to destroy.
```

# 5. Cambiar la variable var.filename a "bye.txt" y aplicar

Cambiar la variable en `variables.tf` o pasarla como parámetro:
```tf
variable "filename" {
  type    = string
  default = "bye.txt"
}
```

Esto provocará el reemplazo del recurso `file`: primero lo destruye y luego lo crea. Este es el comportamiento por defecto en Terraform cuando una modificación exige recrear el recurso.

```vim
~ filename             = "hello.txt" -> "bye.txt" # forces replacement

Plan: 1 to add, 0 to change, 1 to destroy.
```

# 6. Validar resultados: primero destruye y luego crea

En la salida se observará el siguiente orden en la ejecución:
```vim
local_file.file: Destroying...
local_file.file: Destruction complete after 0s
local_file.file: Creating...
local_file.file: Creation complete after 0s
```

# 7. Modificar el bloque "file": content y lifecycle

Modificar `main.tf` para usar la regla de ciclo de vida `create_before_destroy = true`.

```tf
resource "local_file" "file" {
  filename        = var.filename
  file_permission = "0700"
  content         = "Bye..."

  lifecycle {
    create_before_destroy = true
  }
}
```

# 8. Aplicar y validar el resultado: primero crea y luego destruye

Aplicar los cambios. Se observará que el orden de ejecución se invierte:

```vim
~ content              = "Hello World" -> "Bye..." # forces replacement

Plan: 1 to add, 0 to change, 1 to destroy.

local_file.file: Creating...
local_file.file: Creation complete after 0s
local_file.file (deposed object 9f2202c7): Destroying...
local_file.file: Destruction complete after 0s
```

> [!NOTE]
> **Propósito de `create_before_destroy` en la nube**:
> Esta regla es fundamental para lograr **cero tiempo de inactividad (Zero Downtime)** al actualizar infraestructura crítica. Por defecto, Terraform destruye un recurso antes de crear su reemplazo (lo que genera una interrupción temporal del servicio). Con esta regla, el nuevo recurso se despliega y verifica primero, y solo cuando está listo, se procede a eliminar el recurso anterior.
>
> *Ejemplo real: Servidor web detrás de un balanceador de carga.*
> Al actualizar la imagen de sistema operativo del servidor, Terraform creará el nuevo servidor web. Una vez que este esté listo y recibiendo tráfico a través del balanceador, se destruirá el servidor web antiguo, garantizando la continuidad del servicio para los usuarios.

# 9. Modificar el bloque lifecycle para usar prevent_destroy

Modificar el archivo `main.tf` para evitar que el recurso sea destruido accidentalmente:

```tf
resource "local_file" "file" {
  filename        = var.filename
  file_permission = "0700"
  content         = "Bye..."

  lifecycle {
    prevent_destroy = true
  }
}
```

Aplicar este cambio para actualizar el estado (`terraform apply`).

# 10. Forzar el reemplazo del recurso con prevent_destroy activo

Cambiar el permiso del archivo en `main.tf` para forzar su reemplazo:

```tf
resource "local_file" "file" {
  filename        = var.filename
  file_permission = "0777" # Modificado para forzar reemplazo
  content         = "Bye..."

  lifecycle {
    prevent_destroy = true
  }
}
```

Intentar aplicar (`terraform apply`). Se presentará el siguiente error:

```vim
Error: Instance cannot be destroyed

on main.tf line 1:
resource "local_file" "file" {

Resource local_file.file has lifecycle.prevent_destroy set, but the plan requires destroying this resource.
```

> [!IMPORTANT]
> `prevent_destroy` es una medida de seguridad muy útil para bases de datos, redes o recursos críticos de producción, impidiendo que cambios accidentales los eliminen.

# 11. Entender ignore_changes

La tercera regla clave de ciclo de vida es `ignore_changes`. Sirve para ignorar modificaciones externas o cambios en ciertos atributos que no se desea que Terraform sobrescriba o detecte.

11.1. Modificar `main.tf` para ignorar los cambios del atributo `content` y regresar `prevent_destroy` a `false` (o eliminarlo):

```tf
resource "local_file" "file" {
  filename        = var.filename
  file_permission = "0700"
  content         = "Contenido Inicial"

  lifecycle {
    ignore_changes = [
      content,
    ]
  }
}
```

11.2. Aplicar este cambio (`terraform apply`).

11.3. Modificar el valor de `content` en `main.tf`:

```tf
  content         = "Contenido Modificado en Terraform"
```

11.4. Ejecutar `terraform plan`.

Se observará que Terraform indica que no hay cambios pendientes (`No changes. Your infrastructure matches the configuration.`), debido a que está ignorando cualquier cambio en el atributo `content`.

# 12. Renombrar un recurso sin recrearlo (moved)

Cuando se cambia el nombre lógico de un recurso en el código (por ejemplo, cambiar `resource "local_file" "file"` a `resource "local_file" "archivo"`), Terraform por defecto asume que el recurso original fue eliminado y que se debe crear uno nuevo. 

Para indicarle a Terraform que el recurso físico es el mismo y que solo se está renombrando en el código, se utiliza el bloque `moved`.

12.1. Modificar `main.tf` renombrando el recurso de `file` a `archivo` y agregando el bloque `moved` (asegurarse de que `prevent_destroy` esté en `false` o no esté configurado):

```tf
resource "local_file" "archivo" {
  filename        = var.filename
  file_permission = "0700"
  content         = "Contenido Inicial"
}

moved {
  from = local_file.file
  to   = local_file.archivo
}
```

12.2. Ejecutar `terraform plan`.

Se observará en la salida que Terraform detecta el cambio como un traslado (moved) en el estado interno, y no como una destrucción/creación:

```vim
Terraform will perform the following actions:

  # local_file.file has moved to local_file.archivo

Plan: 0 to add, 0 to change, 0 to destroy.
```

12.3. Aplicar el cambio (`terraform apply`). El recurso físico no se modificará, pero Terraform actualizará el archivo de estado con el nuevo nombre lógico.

# 13. Destruir todo

Para poder eliminar toda la infraestructura, es necesario asegurarse de no tener la regla `prevent_destroy` activa en `true`. Si está activa, cambiarla a `false` o eliminarla.

Ejecutar:
```bash
terraform destroy
```

Salida esperada:
```vim
Destroy complete! Resources: 1 destroyed.
```
