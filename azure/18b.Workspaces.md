# 18. Workspaces (Azure) <!-- omit in toc -->

# 1. Crear la carpeta /workspaces y acceder a ella

# 2. Crear dos Workspaces
> Por defecto, Terraform crea un workspace inicial llamado `default`.

```vim
terraform workspace new segundo
terraform workspace new tercero
```
> Created and switched to workspace "segundo"!
>
> Created and switched to workspace "tercero"!

> Nota: Cuando creas nuevos workspaces, Terraform genera automáticamente la carpeta `terraform.tfstate.d` para almacenar el estado aislado de cada uno.

# 3. Seleccionar el workspace "segundo"
```tf
terraform workspace select segundo
```
> Switched to workspace "segundo".

# 4. Listar los workspaces
```vim
terraform workspace list
```
```vim
  default
* segundo
  tercero
```

# 5. Crear terraform.tf (en la raíz)
```tf
terraform {
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 3.0"
    }
  }
}
```

# 6. Crear variables.tf (en la raíz)
```tf
variable "location" {
  type = map(string)
  default = {
    "segundo" = "East US"
    "tercero" = "West US"
  }
}

variable "replication" {
  type = map(string)
  default = {
    "segundo" = "LRS"
    "tercero" = "GRS"
  }
}
```

> Map = Objeto que agrupa valores bajo claves (keys).
>
> [Más info oficial sobre Map Types](https://www.terraform.io/language/expressions/types#map)

# 7. Crear provider.tf (en la raíz)
En AWS la región se configura aquí. En Azure, el provider general solo requiere sus `features`. Las ubicaciones se mandan directamente a los recursos luego.
```tf
provider "azurerm" {
  features {}
}
```


# 8. Crear la carpeta "modules". Dentro de ella, crea los siguientes archivos:

> Estructura: `./workspaces/modules/`

## 8.1. modules/variables.tf
```tf
variable "res_location" {
  type    = string
}

variable "repl_type" {
  type    = string
}
```

## 8.2. modules/storage.tf
```tf
resource "azurerm_resource_group" "rg" {
  name     = "rg-workspace-${terraform.workspace}"
  location = var.res_location
}

resource "azurerm_storage_account" "storage" {
  # El nombre debe ser único globalmente. Interpolaremos el workspace para evitar colisiones.
  name                     = "storage159abc${terraform.workspace}"
  resource_group_name      = azurerm_resource_group.rg.name
  location                 = azurerm_resource_group.rg.location
  account_tier             = "Standard"
  account_replication_type = var.repl_type

  tags = {
    Ambiente = terraform.workspace
  }
}
```
> Nota: `${terraform.workspace}` es una variable de sistema nativa que devuelve el nombre del workspace actual.

## 8.3. modules/output.tf
```tf
output "storage_tier" {
  value = azurerm_storage_account.storage.account_tier
}

output "storage_replication" {
  value = azurerm_storage_account.storage.account_replication_type
}

output "storage_location" {
  value = azurerm_storage_account.storage.location
}

output "resource_group_name" {
  value = azurerm_resource_group.rg.name
}
```

# 9. Regresar a la carpeta principal (`./workspaces`) y crear main.tf
```tf
module "storage_demo" {
  source       = "./modules"
  
  # La función lookup extraerá el valor dependiendo de la key proporcionada 
  # (en este caso, terraform.workspace "segundo" o "tercero")
  res_location = lookup(var.location, terraform.workspace)
  repl_type    = lookup(var.replication, terraform.workspace)
}
```
> [Más información de la función Lookup](https://www.terraform.io/language/functions/lookup)

# 10. Crear output.tf (en la raíz)
```tf
output "storage_tier" {
  value = module.storage_demo.storage_tier
}

output "storage_replication" {
  value = module.storage_demo.storage_replication
}

output "storage_location" {
  value = module.storage_demo.storage_location
}

output "resource_group_name" {
  value = module.storage_demo.resource_group_name
}
```

# 11. Iniciar y aplicar (Estando en el workspace 'segundo')
```vim
terraform init
terraform apply -auto-approve
```

Deberás ver como resultado:
```vim
Outputs:

resource_group_name = "rg-workspace-segundo"
storage_location = "eastus"
storage_replication = "LRS"
storage_tier = "Standard"
```

## Validar Grupo de Recursos en Azure
```vim
az group show --name rg-workspace-segundo --query "{Name:name, Location:location, ProvisioningState:properties.provisioningState}"
```

# 12. Cambiar al workspace "tercero" y aplicar
Ahora saltamos de universo. Este workspace tendrá su propia topología independiente.
```vim
terraform workspace select tercero
```
> Switched to workspace "tercero".

```vim
terraform show
terraform plan
terraform apply -auto-approve
```

Resultado esperado (Anota como cambiaron la región y el tipo de replicación dictados en el map):
```vim
Outputs:

resource_group_name = "rg-workspace-tercero"
storage_location = "westus"
storage_replication = "GRS"
storage_tier = "Standard"
```
> Puedes validar mediante la terminal la existencia del nuevo Resource Group de forma aislada.

```vim
az group show --name rg-workspace-tercero --query "{Name:name, Location:location, ProvisioningState:properties.provisioningState}"
```

# 13. Eliminar los recursos de ambos workspaces aisladamente
Para evitar cargos de nube, limpiaremos el entorno cambiando dinámicamente el apuntador de Terraform:

```vim
# Estando en el workspace tercero destruimos sus componentes:
terraform destroy -auto-approve

# Regresamos la mira al workspace anterior
terraform workspace select segundo

# Y ejecutamos la destrucción de sus componentes:
terraform destroy -auto-approve
```
